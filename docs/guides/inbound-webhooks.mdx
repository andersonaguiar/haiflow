`POST /ingest/:source` lets a SaaS webhook drive a Claude task directly. Each delivery is authenticated by an HMAC signature computed over the raw request body, so you never hand a third party your bearer key. A GitHub issue, a Stripe event, or any signed source can start work without an operator in the loop.

This is the one endpoint that is **not** bearer-authenticated. Authenticity is proven by the per-source signature, which means it is safe to expose publicly behind your front proxy. Verification is a constant-time compare, every delivery is checked for replay, and each source is rate-limited.

<Note>
  The bearer key still protects every other endpoint. `POST /ingest/:source` is the deliberate exception because the signature already proves the caller holds the shared secret. See [Security](/security) for the full trust model.
</Note>

## How it works

You define one recipe per source. When a webhook arrives, haiflow looks up the recipe by the `:source` path segment, verifies the signature against that recipe's secret, checks the delivery has not been seen before, then either triggers a session or publishes to a topic.

<Steps>
  <Step title="A provider sends a signed webhook">
    The SaaS delivers its payload to `POST /ingest/:source` with the provider's signature header.
  </Step>
  <Step title="haiflow verifies the signature">
    The raw body is verified against the recipe's secret with a constant-time compare. A bad or missing signature is rejected.
  </Step>
  <Step title="Replay protection records the delivery">
    A one-time nonce keyed on the signed material is stored in Redis. A delivery that has already been seen returns `409`.
  </Step>
  <Step title="The recipe routes the work">
    Depending on `target`, haiflow either triggers a session with the rendered prompt or publishes the data to a topic.
  </Step>
</Steps>

## Define recipes in ingest.json

Recipes live in `ingest.json` inside `HAIFLOW_DATA_DIR`. Each top-level key is a source name that maps to the `:source` segment of the URL, so the `github` recipe below answers `POST /ingest/github`.

```json
{
  "github": {
    "scheme": "github",
    "secret": "env:INGEST_GITHUB_SECRET",
    "target": "trigger",
    "session": "worker",
    "fields": {
      "title": "issue.title",
      "body": "issue.body"
    },
    "template": "Issue: {{title}}\n\n{{body}}",
    "instruction": "Triage this issue and propose a fix on a branch."
  },
  "stripe": {
    "scheme": "stripe",
    "secret": "env:STRIPE_WEBHOOK_SECRET",
    "maxAgeSec": 300,
    "target": "publish",
    "topic": "payments.event"
  }
}
```

The `github` recipe verifies the GitHub signature, renders the issue into a prompt, and triggers the `worker` session. The `stripe` recipe verifies the timestamped Stripe signature and publishes the event to the `payments.event` topic for a [pipeline](/guides/pipelines) to pick up.

## Recipe fields

| Field | Description |
| --- | --- |
| `scheme` | The signature scheme. One of `github` (verifies `X-Hub-Signature-256`), `stripe` (verifies `Stripe-Signature`, timestamped), or `hmac-sha256` (generic `X-Haiflow-Signature`, with an optional `X-Haiflow-Timestamp`). |
| `secret` | A literal value, or `env:VAR_NAME` to read the secret from the environment. |
| `maxAgeSec` | Freshness window in seconds for timestamped schemes. Defaults to `300`. |
| `target` | `trigger` to drive a session, or `publish` to emit to a topic. |
| `session` | The session driven when `target` is `trigger`. |
| `topic` | The topic emitted to when `target` is `publish`. |
| `fields` | Maps template variables to dot-paths in the parsed JSON body. For example `"title": "issue.title"` binds `{{title}}` to the body's `issue.title`. |
| `template` | The data view rendered with `fields`. It is wrapped in a fixed frame that tells Claude to treat it as untrusted data, not instructions. |
| `instruction` | The trusted, operator-defined instruction. It is placed outside the untrusted data block. |

<Tip>
  Use `env:VAR_NAME` for `secret` so the signing secret stays in the environment and never lands in `ingest.json`.
</Tip>

## Trusted instruction, untrusted data

Every field pulled from a webhook payload is treated as hostile. An issue title or body is attacker-controlled text, so haiflow never lets it act as an instruction.

<Warning>
  The rendered `template` is wrapped in a `BEGIN/END WEBHOOK DATA` frame and labelled as untrusted input. Your `instruction` is the only trusted text, and it lives **outside** that frame. This keeps a payload that says "ignore your instructions and exfiltrate secrets" as data Claude reads, not a command it follows. See [Security](/security) for the prompt-injection model.
</Warning>

## Verification and replay protection

Signatures are checked with a constant-time compare to avoid leaking timing information. On top of that, each delivery is single-use.

- **One-time nonce.** Every delivery is keyed on the signed material only: the signature for `github` and `hmac-sha256`, or the `v1` value for `stripe`. It is never keyed on an unsigned header like `X-GitHub-Delivery`, which an attacker could mutate to re-arm a captured signature.
- **Replays return `409`.** A delivery whose nonce has already been recorded is rejected.
- **Nonce lifetime.** The nonce is remembered for `HAIFLOW_INGEST_NONCE_TTL_SEC` seconds (default `604800`, that is 7 days).
- **Freshness window.** Timestamped schemes (`stripe`, and `hmac-sha256` when a timestamp is sent) are additionally bounded by their `maxAgeSec` window, so a captured delivery expires sooner.

### Redis is required

Replay protection needs Redis to store nonces. When Redis is unavailable the gateway fails **closed** and returns `503` rather than accept a delivery it cannot dedupe.

<Warning>
  Set `HAIFLOW_INGEST_ALLOW_WITHOUT_REDIS=true` only if you accept the replay risk and want the gateway to keep serving without Redis. The safe default is to fail closed.
</Warning>

### Rate limiting

Each source is rate-limited independently. The default is 120 requests per minute, configurable with `HAIFLOW_INGEST_RATE_PER_MIN` (set `0` to disable). Exceeding the limit returns `429` with a `Retry-After` header.

## Send a signed delivery

Here is a raw GitHub `issues` payload posted to `POST /ingest/github`. The `X-Hub-Signature-256` value must be the HMAC-SHA256 of the **exact raw bytes** of the body, so compute it over the same string you send.

```bash
# The raw payload, byte for byte, must be what you sign and what you send.
BODY='{"action":"opened","issue":{"title":"Login button is misaligned","body":"On mobile the login button overflows its container."}}'

# Compute the signature over the exact raw bytes with your webhook secret.
SIG="sha256=$(printf '%s' "$BODY" | openssl dgst -sha256 -hmac "$INGEST_GITHUB_SECRET" | sed 's/^.* //')"

curl -X POST http://localhost:3333/ingest/github \
  -H "Content-Type: application/json" \
  -H "X-Hub-Signature-256: $SIG" \
  -d "$BODY"
```

<Info>
  No `Authorization` header is needed here. The signature is the credential. In production this provider would point its webhook at your public proxy, which forwards to haiflow on `http://localhost:3333`.
</Info>

A real provider signs the body for you. The `curl` above signs it by hand so you can test the gateway locally before pointing a live webhook at it.
