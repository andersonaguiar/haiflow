haiflow runs on a flat Claude Code subscription, so the tasks it orchestrates cost nothing per token. To make that concrete, haiflow records the tokens each task consumed and reports the equivalent amount a per-token API caller would have paid for the same work. Two endpoints expose this: `GET /usage` for aggregate totals since a timestamp, and `GET /usage/window` for the rolling rate-limit windows.

<Note>
The dollar figure is an estimate. It comes from a maintained price table mapped against measured token counts, not a bill. Use it to see roughly what you avoided, not to reconcile an invoice.
</Note>

## What gets measured

Every task that haiflow runs lands in the [task ledger](/concepts/task-ledger) with its token counts. The usage endpoints aggregate those records:

| Field | Meaning |
| --- | --- |
| `inputTokens` | Tokens sent into the model as prompt. |
| `outputTokens` | Tokens the model generated in response. |
| `cacheCreationTokens` | Tokens written into the prompt cache. |
| `cacheReadTokens` | Tokens served from the prompt cache. |
| `totalTokens` | Sum of the above across the matched tasks. |
| `savedUsd` | Estimated per-token API cost you avoided, from the price table. |

## Aggregate usage since a timestamp

`GET /usage` returns totals across all matching tasks. With no parameters it covers the last 24 hours of activity for every session.

<ParamField query="session" type="string">
  Limit the totals to a single session. Omit it to aggregate the whole account.
</ParamField>

<ParamField query="since" type="string (ISO 8601)">
  Count only tasks recorded at or after this timestamp. Defaults to 24 hours ago.
</ParamField>

```bash
curl http://localhost:3333/usage \
  -H "Authorization: Bearer $HAIFLOW_API_KEY"
```

Scope it to one session and a custom start time with the query parameters:

```bash
curl "http://localhost:3333/usage?session=billing-agent&since=2026-06-01T00:00:00Z" \
  -H "Authorization: Bearer $HAIFLOW_API_KEY"
```

The response echoes the window you asked for, then reports the totals:

```json
{
  "since": "2026-06-01T00:00:00Z",
  "session": "billing-agent",
  "tasks": 42,
  "totalTokens": 980000,
  "inputTokens": 120000,
  "outputTokens": 60000,
  "cacheCreationTokens": 300000,
  "cacheReadTokens": 500000,
  "savedUsd": 11.73
}
```

<ResponseField name="since" type="string">
  The start timestamp the totals were computed from.
</ResponseField>
<ResponseField name="session" type="string">
  The session the totals are scoped to, or all sessions when no `session` was given.
</ResponseField>
<ResponseField name="tasks" type="number">
  Number of tasks counted in the window.
</ResponseField>
<ResponseField name="totalTokens" type="number">
  Total tokens consumed across those tasks.
</ResponseField>
<ResponseField name="inputTokens" type="number">
  Prompt tokens sent to the model.
</ResponseField>
<ResponseField name="outputTokens" type="number">
  Tokens generated by the model.
</ResponseField>
<ResponseField name="cacheCreationTokens" type="number">
  Tokens written into the prompt cache.
</ResponseField>
<ResponseField name="cacheReadTokens" type="number">
  Tokens served from the prompt cache.
</ResponseField>
<ResponseField name="savedUsd" type="number">
  Estimated per-token API cost avoided, from the price table.
</ResponseField>

## Rolling rate-limit windows

`GET /usage/window` reports the two windows that govern a Claude Code subscription: a rolling 5 hours and a rolling 7 days. This is the view to watch if you are pushing a lot of work through and want to stay clear of a subscription rate limit.

By default it covers the whole account. Add `?session=` to scope it to one session.

```bash
curl http://localhost:3333/usage/window \
  -H "Authorization: Bearer $HAIFLOW_API_KEY"
```

```bash
curl "http://localhost:3333/usage/window?session=billing-agent" \
  -H "Authorization: Bearer $HAIFLOW_API_KEY"
```

Each window carries its own task count, the full token breakdown, and estimated savings:

```json
{
  "session": "all",
  "windows": {
    "5h": { "tasks": 3, "totalTokens": 41000, "inputTokens": 30000, "outputTokens": 9000, "cacheCreationTokens": 1000, "cacheReadTokens": 1000, "savedUsd": 0.42 },
    "7d": { "tasks": 88, "totalTokens": 1200000, "inputTokens": 900000, "outputTokens": 250000, "cacheCreationTokens": 30000, "cacheReadTokens": 20000, "savedUsd": 14.10 }
  },
  "alertThresholdTokens": null,
  "alert": false
}
```

Each window bucket has the same shape as the [`GET /usage`](/api-reference/introduction) totals: `tasks`, `totalTokens`, `inputTokens`, `outputTokens`, `cacheCreationTokens`, `cacheReadTokens`, and `savedUsd`.

## Usage alerts

Set `HAIFLOW_USAGE_ALERT_TOKENS` to a token count and `GET /usage/window` will flip `alert` to `true` once the 5-hour window crosses that threshold. The configured value is echoed back as `alertThresholdTokens`, so a client can show how close you are. When the variable is unset, `alertThresholdTokens` is `null` and `alert` is always `false`.

<Warning>
The alert is informational only. It never throttles, queues, or blocks work. It surfaces a warning so you can ease off before you hit a subscription rate limit, nothing more.
</Warning>

You can poll `GET /usage/window` and react when `alert` becomes `true`:

```bash
curl "http://localhost:3333/usage/window?session=all" \
  -H "Authorization: Bearer $HAIFLOW_API_KEY"
```

See [Configuration](/configuration) for where to set `HAIFLOW_USAGE_ALERT_TOKENS`.
