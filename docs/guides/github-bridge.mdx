Mention `@haiflow` in a GitHub issue or pull request comment and Claude Code addresses it in the repo you have checked out locally. It works on a branch, opens a draft PR, and never touches the default branch. You stay in GitHub, Claude does the work in the session.

The bridge itself is deliberately thin. It is a gated relay: it verifies the webhook, checks two allowlists, and forwards the comment to a haiflow session. Claude handles the branch, commits, and PR itself, because it already has `gh` and `git` available inside the session.

## How it works

The bridge is a small HTTP listener that sits between GitHub and a haiflow session. It does the gatekeeping, then hands off.

<Steps>
  <Step title="Someone comments on an issue or PR">
    The comment includes the trigger phrase, `@haiflow` by default.
  </Step>
  <Step title="GitHub delivers a webhook">
    GitHub posts the event to the bridge, signed with the `X-Hub-Signature-256` HMAC.
  </Step>
  <Step title="The bridge verifies and gates the event">
    It checks the signature against `GITHUB_WEBHOOK_SECRET`, then confirms the repo and sender both match the allowlists. If anything fails, the event is refused.
  </Step>
  <Step title="The comment is forwarded to a session">
    The comment text is relayed to the session named by `GITHUB_SESSION`, whose cwd is the cloned repo.
  </Step>
  <Step title="Claude does the Git work">
    Claude creates a branch, makes the changes, commits, and opens a draft PR with `gh` and `git`. It never pushes the default branch.
  </Step>
  <Step title="The result is posted back">
    When `GITHUB_POST_COMMENTS` is on, Claude's summary is posted back to the thread as a GitHub comment.
  </Step>
</Steps>

<Note>
  The session that the bridge drives must have its cwd set to the cloned repository. Point `GITHUB_SESSION` at that session. See [Sessions](/concepts/sessions) for how a session's working directory is set.
</Note>

## Running the bridge

The bridge is a separate process from the haiflow server. Start it once the target session is up and its cwd is the cloned repo.

<CodeGroup>

```bash haiflow CLI
haiflow github
```

```bash bun
bun run github
```

</CodeGroup>

It listens on `GITHUB_PORT` (default `3334`) and calls back into the haiflow server at `HAIFLOW_URL` to drive the session.

On the GitHub side, add a webhook to the repository: set the payload URL to the bridge's address and port, set the content type to `application/json`, and set the secret to the same value as `GITHUB_WEBHOOK_SECRET` so the signature verifies. Subscribe it to issue and pull request comment events.

## Configuration

Configure the bridge with environment variables. Both allowlists must be populated for the bridge to act on anything.

| Variable | Default | Description |
| --- | --- | --- |
| `HAIFLOW_URL` | `http://localhost:$PORT` | Base URL the bridge calls to drive the haiflow server. |
| `GITHUB_WEBHOOK_SECRET` | (required) | Shared secret used to verify the `X-Hub-Signature-256` HMAC on every delivery. |
| `GITHUB_ALLOWED_REPOS` | (required to act) | Comma-separated list of repos the bridge will act on. |
| `GITHUB_ALLOWED_SENDERS` | (required to act) | Comma-separated list of GitHub usernames allowed to trigger the bridge. |
| `GITHUB_SESSION` | `default` | The session the bridge drives. Its cwd should be the cloned repo. |
| `GITHUB_PORT` | `3334` | Port the bridge listens on for GitHub webhooks. |
| `GITHUB_TRIGGER_PHRASE` | `@haiflow` | The phrase a comment must contain to trigger the bridge. |
| `GITHUB_POST_COMMENTS` | `true` | Post Claude's summary back to the thread as a GitHub comment via `gh`. |

```bash .env
GITHUB_WEBHOOK_SECRET=your-long-random-secret
GITHUB_ALLOWED_REPOS=your-org/your-repo
GITHUB_ALLOWED_SENDERS=your-username
GITHUB_SESSION=repo-agent
```

<Tip>
  See [Configuration](/configuration) for how haiflow loads environment variables and how `$PORT` resolves for `HAIFLOW_URL`.
</Tip>

## Trust boundary

<Warning>
  The two allowlists are the trust boundary. With either `GITHUB_ALLOWED_REPOS` or `GITHUB_ALLOWED_SENDERS` empty, every webhook is refused. Anyone who can comment on an allowlisted repo can drive Claude in your session, so keep both lists tight.

  The comment text is treated as untrusted input: it is wrapped in a data frame, and Claude is instructed to open a draft PR and never push the default branch. Even so, review every PR the bridge opens before you merge it.
</Warning>

Because the bridge acts only when the signature verifies and both allowlists match, an attacker would need a valid signed delivery from an allowlisted repo plus an allowlisted sender to reach Claude. That is why the secret and the two lists are the only things standing between a public comment box and code running in your session. For the broader hardening model, see [Security](/security).

## Using it

Once the bridge is running and the webhook is configured, drop a comment on any issue or PR in an allowlisted repo:

```text
@haiflow the date parser drops the timezone offset on ISO strings.
Add a failing test, fix it, and open a draft PR.
```

The bridge forwards the comment to your session. Claude branches off the default branch, makes the change, and opens a draft PR. If `GITHUB_POST_COMMENTS` is on, it replies in the thread with a summary and a link to the PR. The default branch is never written to directly.

## Known limitation

The standalone bridge has no replay nonce of its own. It relies on queue-level dedup to avoid acting on a redelivered webhook twice. That is enough for most setups, but it is not persistent replay protection.

<Info>
  If you need durable replay protection, use the `/ingest github` recipe instead of the standalone bridge. It records a nonce so a replayed delivery is rejected even across restarts. See [Inbound webhooks](/guides/inbound-webhooks).
</Info>
